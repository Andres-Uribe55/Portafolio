El Desafío

La mayoría de los ejecutores de tareas tienen dificultades para escalar horizontalmente. El desafío principal fue construir un sistema que pudiera manejar miles de tareas concurrentes de larga duración sin un solo punto de falla.

Las soluciones internas anteriores sufrían de condiciones de carrera frecuentes durante el reequilibrio de trabajadores y alta latencia al distribuir trabajos en múltiples zonas de disponibilidad. Necesitábamos una arquitectura sin líder que pudiera mantener una consistencia fuerte asegurando una disponibilidad del 99.99%.

verified Arquitectura

La Solución

Implementamos un protocolo de chismes (gossip protocol) entre pares para la sincronización de estados. Al utilizar una tabla de hash distribuida (DHT), Nebula distribuye las cargas de trabajo de manera uniforme en el clúster sin un nodo maestro central.

  • check_circle Nodos de trabajo con capacidad de recuperación automática mediante controles de salud.
  • check_circle Escalado dinámico basado en la profundidad de la cola y la presión de la CPU.
  • check_circle Replicación multi-región con tipos de datos replicados libres de conflictos (CRDTs).

Profundización Técnica

A continuación se presenta la implementación de nuestra lógica de planificador de tareas personalizada que utiliza hash consistente para minimizar las reasignaciones durante la rotación de nodos.

scheduler.go 
func (s *Scheduler) AssignTask(taskID string) (string, error) {
    // Buscar el nodo óptimo usando hash consistente
    node, err := s.ring.GetNode(taskID)
    if err != nil {
        return "", err
    }
    // Verificación atómica check-and-set para propiedad del trabajo
    success, err := s.kv.SetNX(ctx, "lock:"+taskID, node.ID, time.Second*30)
    if !success || err != nil {
        return "", ErrTaskAlreadyClaimed
    }
    return node.Address, nil
}
